SYLUXENT ERP SYSTEM ARCHITECTURE

1. System Overview

Syluxent is a web-based Enterprise Resource Planning system built with Flask.
It provides modules for authentication, dashboard monitoring, client records,
sales orders, invoices, purchase orders, reports, analytics, evaluation forms,
and administrator database controls.

The system follows a standard web application architecture:

Browser Frontend
    -> Flask Web Application
    -> Service and Business Logic Layer
    -> SQLAlchemy ORM
    -> Database Storage

For local development, the system can use SQLite automatically.
For Render production deployment, the system should use Supabase PostgreSQL
through the DATABASE_URL environment variable.


2. Main Components

2.1 Browser / Frontend Layer

Component name:
    Browser Client

Technology:
    HTML, CSS, JavaScript, Flask Jinja templates

Main folders/files:
    templates/
    static/css/styles.css
    static/css/theme-overrides.css
    static/js/system-states.js

Responsibilities:
    - Displays login, dashboard, admin, sales, invoices, purchase orders,
      reports, analytics, profile, and evaluation pages.
    - Sends form submissions and file uploads to Flask routes.
    - Receives HTML pages or JSON responses from the backend.
    - Uses CSS and JavaScript for layout, theme, and UI state behavior.

Connection:
    Browser connects to the deployed Render URL over HTTPS.


2.2 Flask Application Layer

Component name:
    Flask Web App

Technology:
    Flask
    Flask-SQLAlchemy
    Werkzeug security
    Gunicorn for production

Main file:
    app.py

Responsibilities:
    - Creates the Flask app instance.
    - Configures database connection.
    - Defines SQLAlchemy database models.
    - Defines web page routes and JSON API routes.
    - Handles login, registration, logout, session state, and role checks.
    - Coordinates sales, invoice, purchase order, reports, analytics, and admin
      workflows.
    - Initializes database tables through db.create_all() inside app.app_context().

Important production entry point:
    gunicorn app:app

Connection:
    Render starts the Flask app through Gunicorn.
    Browser requests are routed to Flask routes.
    Flask uses SQLAlchemy to communicate with the configured database.


2.3 Service Layer

Component name:
    Business Logic and Services

Main files:
    analytics_services.py
    admin_services.py

Responsibilities:
    analytics_services.py:
        - Builds analytics dashboard payloads.
        - Calculates overview KPIs.
        - Produces sales trends, client analysis, expense breakdown,
          sales analysis, and comparative analytics.
        - Supports Excel workbook preview for analytics uploads.

    admin_services.py:
        - Supports admin data grid operations.
        - Supports CSV export.
        - Provides database health and schema inspection.
        - Provides safe SQL and maintenance helpers for the admin interface.

Connection:
    Flask route handlers call service functions.
    Service functions query or update data through SQLAlchemy sessions.


2.4 ORM / Data Access Layer

Component name:
    SQLAlchemy ORM

Technology:
    Flask-SQLAlchemy
    SQLAlchemy

Main file:
    app.py

Responsibilities:
    - Maps Python model classes to database tables.
    - Handles database sessions, queries, inserts, updates, and deletes.
    - Protects against raw SQL injection for normal ORM operations.
    - Allows the same application logic to work with SQLite locally and
      PostgreSQL in production.

Connection:
    Flask app -> SQLAlchemy ORM -> SQLite or Supabase PostgreSQL


2.5 Database Layer

Component name:
    Database Storage

Local development:
    SQLite database file:
        database.db

Production:
    Supabase PostgreSQL

Production environment variable:
    DATABASE_URL

Example production format:
    postgresql+psycopg2://USER:PASSWORD@HOST:PORT/postgres?sslmode=require

Responsibilities:
    - Stores user accounts, roles, client records, sales orders, invoices,
      purchase orders, analytics records, audit logs, sessions, password reset
      requests, and evaluation responses.
    - Provides persistent cloud storage when configured through Supabase.

Connection:
    Render provides DATABASE_URL to the Flask app.
    Flask-SQLAlchemy uses that value to create the database engine.
    SQLAlchemy connects to Supabase PostgreSQL through psycopg2.


2.6 Deployment Layer

Component name:
    Render Web Service

Technology:
    Render free-tier web service
    Gunicorn

Main file:
    render.yaml

Build command:
    pip install -r requirements.txt

Start command:
    gunicorn app:app

Required Render environment variables:
    DATABASE_URL
    SECRET_KEY
    DEFAULT_ADMIN_PASSWORD

Optional Render environment variables:
    DEFAULT_ADMIN_USERNAME
    DEFAULT_MANAGER_PASSWORD

Responsibilities:

    - Provides runtime environment variables.
    - Serves the application over HTTPS.

Connection:
    GitHub repository -> Render build/deploy pipeline -> Gunicorn -> Flask app

4. Main Table Relationships

roles
    one role -> many users

users
    one user -> many session_records
    one user -> many audit_logs
    one user -> many password_resets

clients
    one client -> many client_aliases
    one client -> many sales_orders

sales_orders
    one sales order -> many sales_order_items
    one sales order -> many invoices

purchase_orders
    one purchase order -> many purchase_order_debits

evaluation_sessions
    one evaluation session -> many evaluation_responses

evaluation_questions
    one evaluation question -> many evaluation_responses

analytics_data
    stores analytics ledger records using logical source fields.
    It does not require enforced foreign keys to source tables.

6. Main Application Workflows

6.1 Login Workflow

Browser
    -> POST /login
    -> Flask validates username and password hash
    -> Flask checks user status and role
    -> Session data is created
    -> Session record is inserted
    -> User is redirected to dashboard


6.2 Sales Order Workflow

Browser
    -> Sales order page or Excel upload
    -> Flask parses uploaded or form data
    -> Client matching identifies official client records
    -> SalesOrder and SalesOrderItem records are created
    -> Client financial totals are refreshed
    -> Dashboard and reports can read updated records


6.3 Invoice Workflow

Browser
    -> Invoice page or create invoice API
    -> Flask validates invoice fields
    -> Invoice record is created or updated
    -> Payment status, balance, and tax fields are calculated
    -> Client financial totals are refreshed


6.4 Purchase Order / Expense Workflow

Browser
    -> Purchase order page or upload workflow
    -> Flask validates expense and debit data
    -> PurchaseOrder record is created
    -> Related PurchaseOrderDebit records are attached
    -> Expense reports and analytics can read updated data


6.5 Analytics Workflow

Operational records or historical uploads
    -> Analytics generation or upload endpoints
    -> analytics_data ledger records
    -> analytics_services.py calculations
    -> JSON analytics API responses
    -> Analytics dashboard visual output


6.6 Admin Database Interface Workflow

Admin browser session
    -> Protected admin route
    -> Role check confirms admin access
    -> admin_services.py handles schema, grid, export, health, or maintenance
    -> SQLAlchemy performs database operations
    -> Admin receives JSON or downloadable response


7. Production Connection Flow

Step 1:
    User opens the Render application URL in the browser.

Step 2:
    Render routes the HTTPS request to Gunicorn.

Step 3:
    Gunicorn sends the request to the Flask app object in app.py.

Step 4:
    Flask route handler processes the request.

Step 5:
    If data is needed, Flask uses SQLAlchemy.

Step 6:
    SQLAlchemy uses DATABASE_URL to connect to Supabase PostgreSQL.

Step 7:
    Supabase returns query results.

Step 8:
    Flask returns an HTML page, JSON response, CSV export, or redirect.


8. Local Development Connection Flow

Step 1:
    Developer runs:
        python app.py

Step 2:
    If DATABASE_URL is not set, Flask uses:
        sqlite:///database.db

Step 3:
    db.create_all() creates missing local SQLite tables.

Step 4:
    Developer opens the local Flask URL and tests the system.


9. Important Environment Variables

DATABASE_URL:
    Production database connection string.
    Used by Render to connect Flask to Supabase PostgreSQL.
    Must not be committed to GitHub.

SECRET_KEY:
    Flask secret key for sessions.
    Must be private in production.

DEFAULT_ADMIN_PASSWORD:
    Used to seed the first admin user when no admin exists.
    Must be private.

DEFAULT_ADMIN_USERNAME:
    Optional. Defaults to admin.

DEFAULT_MANAGER_PASSWORD:
    Optional. Seeds a manager account if set.

DEFAULT_STAFF_PASSWORD:
    Optional. Seeds a staff account if set.


10. Render Configuration

12. Key Reliability Notes

SQLite:
    Good for local development.
    Not reliable for persistent storage on Render free tier because local files
    can be lost across restarts or redeploys.

Supabase PostgreSQL:
    Recommended for production persistence.
    Data remains in the cloud database across Render restarts.

Render:
    Free services may sleep after inactivity.
    First request after sleep may be slower.

db.create_all():
    Creates missing tables from SQLAlchemy models.
    Runs inside app.app_context().
    Must run only after all models are declared.
    It does not replace a full migration system for future production schema
    changes.

15. Interaction Summary

Admin:
    Controls the system configuration, user access, database interface,
    audit trail, exports, and administrative visibility.

Manager:
    Consumes business intelligence generated from operational records and
    historical analytics data.

Staff:
    Creates and updates the operational records that feed invoices, client
    financials, purchase tracking, reporting, and analytics.

System:
    Uses Flask route guards to enforce role permissions, service modules to
    organize business logic, SQLAlchemy to manage data access, and Supabase
    PostgreSQL to persist production data.

        Expense tables
        Analytics/evaluation tables

    Under deployment:
        GitHub main -> Render -> Gunicorn -> Flask app
        Render Environment Variables -> Flask app
        DATABASE_URL -> Supabase


16.10 Short Explanation for Presentation

Use this explanation when presenting the diagram:

    "The system starts with three user roles: Admin, Manager, and Staff. Each
    role accesses the application through the browser, but route guards in
    Flask control which modules they can use. Staff users create operational
    records such as sales orders, invoices, and purchase orders. Managers read
    those records through dashboards, reports, and analytics services. Admins
    manage users, database tools, exports, schema views, and audit logs. All
    route handlers use service logic and SQLAlchemy models to query or update
    the database. In production, the database connection comes from Render's
    DATABASE_URL environment variable and points to Supabase PostgreSQL. In
    local development, if DATABASE_URL is absent, the same app uses SQLite
    database.db."

17.5 Simplified Presentation Explanation

Use this short explanation:

    "The user interacts with the front end through the browser. The front end
    sends requests, forms, uploads, and actions to the back end. The back end is
    the Flask server that validates the request, checks the user's role, applies
    business logic, and communicates with the database through SQLAlchemy. The
    database stores records such as users, clients, sales orders, invoices,
    purchase orders, analytics data, and audit logs. After the database returns
    the needed data, the back end prepares the response and sends it back to the
    front end. The front end then displays the updated page, dashboard, report,
    or message to the user."
